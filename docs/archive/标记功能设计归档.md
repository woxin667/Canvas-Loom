## 设计基础

标记功能成立的前提是：Obsidian Canvas 的 `.canvas` 文件本质上是 JSON，并且会保留节点上的未知自定义字段。

这意味着插件可以把标记直接存到节点数据中，例如：

- `badge`
- `badgeType`

这样做的收益是：

- 数据直接随 `.canvas` 持久化
- 重启后不会丢失
- 分享画布文件时标记也能一起带走
- 不需要额外的数据库或侧边存储

## 写入流程

用户设置或修改标记时，核心流程是：

1. 打开标记输入模态框
2. 识别输入内容的类型：
   - 数字
   - 文字
   - emoji
3. 更新当前节点的 DOM 属性，提供即时反馈
4. 读取 `canvas.getData()`
5. 在目标节点上写入 `badge` 和 `badgeType`
6. 调用 `canvas.setData()` 和 `requestSave()`

## 读取与恢复流程

当 Canvas 被打开或布局完成后，插件会：

1. 读取当前 Canvas 数据
2. 遍历节点
3. 找出带有标记字段的节点
4. 把标记内容回填到节点 DOM 的 `data-` 属性
5. 交给 CSS 伪元素渲染

## 显示方式

标记显示主要依赖样式层：

- 通过 `data-badge` 提供显示内容
- 通过 `data-badge-type` 区分样式
- 使用 CSS 伪元素显示在卡片右上角

不同类型的视觉规则：

- 数字：偏顺序和优先级
- 文字：偏状态或标签
- emoji：偏视觉化标记

## 组件职责

历史文档中反复出现的职责划分，收敛后可以概括为：

- `BadgeCommands`
  负责入口命令和右键菜单接入
- `BadgeModal`
  负责输入界面
- `BadgeService`
  负责类型识别、数据读写和 DOM 回填
- `BadgeStyles`
  负责样式注入和显示规则

## 当前仍有价值的结论

- 标记应继续依附 `.canvas` 节点数据，而不是外部配置文件
- DOM 立即反馈 + 文件持久化 是这项能力的关键体验设计
- 标记能力本质上是“数据层写入 + 样式层显示”的组合，不需要复杂的状态系统

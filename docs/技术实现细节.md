# 技术实现细节

本文档只说明当前仍在使用的结构和主链路，避免和历史方案混在一起。

## 目录结构

- `src/main.ts`
  插件入口，负责初始化服务、注册命令和菜单。
- `src/adapters/`
  封装 Canvas、剪贴板、存储和文稿创建等外部交互。
- `src/domain/`
  放置卡片、标记模型和排序策略。
- `src/services/`
  负责拆分、标记、内容排序、拼合输出和工作台状态。
- `src/presentation/`
  放置命令、弹窗、视图和样式。
- `src/settings/`
  管理设置类型和设置页。

## 当前核心能力

- 单卡片复制
- 多卡片一键复制、一键拼合和预览工作台
- 单卡片拆分
- 标记和属性管理

## 多卡片主链路

多卡片相关功能已经统一为一条内容处理链路：

1. 从 Canvas 读取当前选区
2. 生成选区快照
3. 按排序模式得到有序卡片列表
4. 生成最终文本
5. 输出到剪贴板、Canvas 卡片、工作台或 Markdown 文稿

相关核心文件：

- `src/services/ContentService.ts`
- `src/services/MergeService.ts`
- `src/services/PreviewWorkbenchService.ts`
- `src/presentation/views/MergeWorkbenchView.ts`

## 为什么使用选区快照

- 让工作台和实时选区解耦
- 工作台被拖成独立窗口后仍可继续使用
- 打开工作台后，后续改动当前选区不会污染已有状态

快照至少包含：

- 节点 `id`
- 文本内容
- 坐标和尺寸
- 标记信息

## 排序实现

自动排序通过策略模式完成：

- `PositionSort`
- `BadgeSort`

工作台额外维护 `manualOrderIds`，用于在自动排序结果基础上继续手动微调。

## 工作台实现

工作台基于 Obsidian `ItemView`，由以下部分组成：

- 顶部工具栏：排序模式、Canvas 信息、卡片数量
- 中部顺序列表：手动模式下可拖拽
- 底部预览与输出区：默认折叠，展开后再生成文本

预览区的主要性能策略：

- 默认折叠
- 展开后才计算内容
- 预览更新带防抖

## 输出实现

- 剪贴板：通过 `ClipboardAdapter` 统一提示
- 新建 Canvas 卡片：以锚点卡片为参考，在右侧创建合并卡片
- 新建 Markdown 文稿：由 `VaultAdapter` 在当前 `.canvas` 同目录生成文件

工作台创建新卡片时会优先尝试恢复源 Canvas，而不是要求原 Canvas 必须保持激活。

## 设置项

当前主要设置项：

- `canvasCardDelimiter`
- `sortPriority`
- `enableBadges`
- `defaultSortMode`

`StorageAdapter` 负责设置读取、保存和旧字段迁移。

## 标记能力

当前标记能力采用“单字段 + 数值友好排序”的实现：

- 只在 `.canvas` 节点上保存 `badge`
- 输入规则：`/^\d+(\.\d+)*$/`
- 排序规则：`localeCompare(undefined, { numeric: true })`
- 无标记卡片在标记排序时排在最后，组内仍按当前位置排序

`enableBadges` 现在是完整开关：

- 关闭时不再给节点补 `data-badge` 显示
- 跳过标记加载
- 清理 DOM 上的 `data-badge`
- 不需要运行时注入或移除样式

## 官方规范相关约束

当前代码结构额外遵守以下约束，这不是临时修补，而是后续开发默认边界：

- 插件样式统一放在根目录 `styles.css`
- 不运行时创建 `<style>` 节点
- 不直接使用 `innerHTML` / `outerHTML`
- UI 状态优先通过 class 切换表达
- Canvas 相关运行时对象通过显式类型收口，避免主链路 `any`
- Promise 在事件回调里要么显式处理，要么使用 `void` 表示有意忽略
- 获取当前 Canvas 上下文时避免依赖已废弃字段，优先使用当前 view 相关 API

这样做的原因不是一次性提审处理，而是这样才符合 Obsidian 官方规范，也更利于后续维护和版本兼容。

## 兼容项

部分旧命令和旧弹窗仍保留在命令面板链路中，用于兼容快捷键和过渡使用，但它们已经不再是主菜单入口。

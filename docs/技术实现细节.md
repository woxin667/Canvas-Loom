# 项目技术实现细节

本文档详细阐述 `obsidian-canvas-card-actions` 插件的最终技术实现方案。

## 1. 项目结构

经过重构后，项目采用分层架构，核心文件及其作用如下：

### 核心文件
- **`src/main.ts`**: 插件的主入口文件。负责插件的生命周期管理、依赖注入、服务初始化和事件注册。
- **`src/settings/CanvasCardActionsSettingTab.ts`**: 负责构建插件的设置页面 UI。
- **`src/settings/ICanvasCardActionsSettings.ts`**: 定义了插件设置对象的 TypeScript 接口。

### 分层架构
- **`src/adapters/`**: 适配器层，包含与外部系统的交互逻辑
  - `CanvasAdapter.ts`: Canvas 操作适配器
  - `ClipboardAdapter.ts`: 剪贴板操作适配器
  - `StorageAdapter.ts`: 数据存储适配器
- **`src/domain/`**: 领域层，包含业务模型和策略
  - `models/`: 数据模型（Card, Badge, CanvasData）
  - `strategies/`: 排序策略（PositionSort, BadgeSort）
- **`src/services/`**: 服务层，包含业务逻辑
  - `CardService.ts`: 卡片管理服务
  - `BadgeService.ts`: 徽章管理服务
  - `ContentService.ts`: 内容处理服务
- **`src/presentation/`**: 表现层，包含UI组件和命令
  - `commands/`: 命令处理器
  - `modals/`: 模态框组件
  - `styles/`: 样式管理器

- **`manifest.json`**: Obsidian 插件的清单文件。定义了插件的ID、名称、版本、描述和作者等元数据。

- **`esbuild.config.mjs`**: 项目的构建脚本。使用 `esbuild` 将 `src` 目录下的 TypeScript 文件打包成单个的 `main.js`，并负责在生产构建时将 `manifest.json` 等静态文件复制到 `build` 目录中。

## 2. 核心功能实现

插件采用依赖注入的方式组织各个服务，在 `main.ts` 的 `onload` 方法中进行初始化和注册。

### 事件监听

- 通过 `CommandRegistry` 服务统一管理所有命令的注册
- 监听 `canvas:node-menu` 和 `canvas:selection-menu` 事件来添加右键菜单项
- 事件处理逻辑被封装在各个Command类中，遵循单一职责原则

### 命令处理

- **`OpenSplitCardModalCommand`**: 打开拆分卡片选择器
- **`SplitCardCommand`**: 保留按分隔符拆分能力的命令封装
- **`CopySingleCardCommand`**: 处理单个卡片复制
- **`CopyByPositionCommand`**: 处理按位置排序复制
- **`CopyByBadgeOrderCommand`**: 处理按徽章顺序复制
- **`OpenBadgeModalCommand`**: 处理徽章模态框打开

每个命令都实现了 `ICommand` 接口，确保一致的处理方式。

### 卡片拆分与创建逻辑

当前版本将拆分能力收拢为统一入口 `拆分卡片...`，但拆分策略仍由 `CardService` 负责。

#### 菜单与选择器

1.  `main.ts` 在 `canvas:node-menu` 中为文本卡片注册 `拆分卡片...`。
2.  点击后执行 `OpenSplitCardModalCommand`，打开 `SplitCardModal`。
3.  `SplitCardModal` 会根据当前卡片内容动态生成可选项：
    - 始终显示 `按分隔符拆分`
    - 始终保留 `按标题拆分` 能力提示
    - 存在可用标题层级时，展开为具体的 `按一级标题拆分`、`按二级标题拆分` 等选项
4.  不可执行的拆分方式会以禁用状态显示，而不是直接隐藏。

#### 按分隔符拆分

1.  调用 `CardService.splitCard(node, delimiter)`。
2.  检查当前文本是否包含设置中的分隔符。
3.  使用 `getDelimitedParts()` 拆分、清理并过滤空内容。
4.  将第一段写回原卡片，其余部分创建为新卡片。

#### 按标题层级拆分

1.  调用 `CardService.splitCardByHeadingLevel(node, level)`。
2.  使用 `getAvailableHeadingSplitOptions()` 分析当前卡片可用的标题层级和预计拆分数量。
3.  使用 `getHeadingSplitParts()` 按指定层级解析 Markdown 标题。
4.  仅将选定层级视为卡片边界，更深层标题内容保留在所属卡片中。
5.  标题前的导言内容会并入第一张匹配标题的卡片。

#### 共用创建流程

1.  两种拆分方式最终都进入 `applySplit()`。
2.  `applySplit()` 先更新原卡片文本，再批量创建新卡片。
3.  新卡片继承原卡片的宽高，并以固定间距排列在右侧。
4.  完成后调用 `canvasAdapter.requestSave()` 持久化。

## 3. 设置页面实现 (`CanvasCardActionsSettingTab.ts`)

- `display()` 方法负责渲染设置页面的所有内容。
- `containerEl.empty()` 确保每次显示时都清空旧的设置，避免重复渲染。
- `new Setting(containerEl)` 创建一个新的设置项。
- `.setName()` 和 `.setDesc()` 设置了选项的中文标题和描述。
- `.addText()` 创建一个文本输入框，用于让用户输入自定义的分隔符。
- `.setValue()` 将输入框的当前值设置为从 `this.plugin.settings` 中读取的值。
- `.onChange()` 注册一个回调函数，当用户修改输入框内容时，将新值赋给 `this.plugin.settings.canvasCardDelimiter`，并立即调用 `this.plugin.saveSettings()` 将其持久化保存。

## 4. 徽章功能实现 (`main.ts`)

插件的徽章功能通过以下技术实现：

### 数据持久化存储

- 利用 Obsidian Canvas API 的 `getData()` 和 `setData()` 方法，将徽章数据直接存储在 `.canvas` 文件中
- 徽章信息以自定义字段形式保存在节点数据中，包括 `badge`（徽章内容）和 `badgeType`（徽章类型）
- 通过 `canvas.requestSave()` 方法将修改持久化到文件

### 徽章类型识别

- **数字徽章**：纯数字内容（如 "1", "3", "99"），显示为圆形蓝色徽章
- **文字徽章**：文字内容（如 "Done", "Important"），显示为圆角矩形灰色徽章
- **Emoji 徽章**：Emoji 表情（如 "✅", "🔥", "⭐"），显示为透明背景的徽章

### 样式实现

- 通过动态注入 CSS 样式，使用 `::after` 伪元素在卡片右上角显示徽章
- 使用 `data-badge` 和 `data-badge-type` 属性控制徽章内容和样式
- 采用高特异性选择器和 `!important` 确保样式正确应用

### 事件处理

- 监听 `canvas:node-menu` 和 `canvas:selection-menu` 事件，在右键菜单中添加徽章选项
- 监听 `layout-change` 和 `file-open` 事件，确保样式在界面变化时持续生效
